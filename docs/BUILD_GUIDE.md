# Backend Build & Swagger Testing Guide (GymPro Backend)

**Stack:** TypeScript · Node.js (LTS) · Express.js · PostgreSQL · Drizzle ORM · Zod · Swagger (OpenAPI)

**Repo:** `https://github.com/samitad2024/Gympro_backend.git`

**Goal:** Build the Node.js/TypeScript backend fully on its own, in a clean, modular, layered architecture, with every endpoint documented in Swagger and tested directly in Swagger UI — before writing a single line of dashboard or mobile code. Development proceeds **step by step**: plan a resource, build it in small pieces, commit each piece separately (small, atomic, conventional commits — like a senior dev, not one giant commit), push, then move to the next resource.

---

## 0. Working principles (read before coding)

1. **Plan before you build.** For every resource: sketch the table shape, the endpoints, and the request/response schema — *before* writing code.
2. **Build in vertical slices.** One resource at a time, fully working end-to-end (schema → repository → service → controller → route → swagger → test), before starting the next.
3. **Small, atomic commits.** Never batch unrelated changes into one commit. Each commit should represent one logical step (e.g. "add drizzle schema for members", "add members repository", "add members zod validators"). Use Conventional Commits (see §9).
4. **Push often.** Push after each meaningful, working commit — not just at the end of the day.
5. **No dead code / no speculative abstractions.** Only build what the current resource needs.

## 4. Clean, modular folder architecture

The project follows a **layered, module-per-resource** architecture. Each resource (members, plans, payments, etc.) is self-contained, so the codebase stays easy to navigate and extend as it grows.

```
Gympro_backend/
├── src/
│   ├── index.ts                  # app bootstrap only
│   ├── app.ts                    # express app wiring (middleware, routes, swagger)
│   ├── config/
│   │   ├── env.ts                # typed, validated env (via zod)
│   │   └── swagger.ts            # swagger-jsdoc config
│   ├── db/
│   │   ├── client.ts             # drizzle + postgres client
│   │   ├── schema/
│   │   │   ├── members.schema.ts
│   │   │   ├── plans.schema.ts
│   │   │   ├── payments.schema.ts
│   │   │   ├── expenses.schema.ts
│   │   │   ├── services.schema.ts
│   │   │   ├── content.schema.ts
│   │   │   ├── notifications.schema.ts
│   │   │   └── index.ts          # re-exports all tables
│   │   └── migrations/           # generated by drizzle-kit
│   ├── middleware/
│   │   ├── error-handler.ts
│   │   ├── validate-request.ts   # generic zod-validation middleware
│   │   └── not-found.ts
│   ├── modules/
│   │   ├── members/
│   │   │   ├── members.routes.ts
│   │   │   ├── members.controller.ts
│   │   │   ├── members.service.ts
│   │   │   ├── members.repository.ts
│   │   │   ├── members.validators.ts   # zod schemas
│   │   │   └── members.types.ts
│   │   ├── plans/            (same 6-file shape)
│   │   ├── payments/         (same 6-file shape)
│   │   ├── expenses/         (same 6-file shape)
│   │   ├── services/         (same 6-file shape)
│   │   ├── content/          (same 6-file shape)
│   │   ├── notifications/    (same 6-file shape)
│   │   └── reports/          (same 6-file shape, built last)
│   └── utils/
│       ├── async-handler.ts
│       └── api-response.ts
├── .env
├── .env.example
├── drizzle.config.ts
├── tsconfig.json
└── package.json
```

**Layer responsibilities (why it's clean):**
- **routes** — only maps HTTP verb + path → controller function, plus the Swagger JSDoc comment.
- **controller** — parses `req`, calls the service, shapes the HTTP response. No business logic, no SQL.
- **service** — business logic (e.g. "confirm payment activates a member and calculates `end_date`"). Framework-agnostic — no `req`/`res` here.
- **repository** — the only layer that talks to Drizzle/the DB.
- **validators** — Zod schemas for request bodies/params/query, used by the `validate-request` middleware.
- **types** — shared TypeScript types/interfaces for the module.

This means: swapping the DB layer, adding a CLI, or writing unit tests only ever touches one layer at a time.

```typescript
  schema: "./src/db/schema/index.ts",

```

`src/db/client.ts`:
```typescript
import { drizzle } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import { env } from "@config/env";
import * as schema from "./schema";

const queryClient = postgres(env.DATABASE_URL, { ssl: "require" });
export const db = drizzle(queryClient, { schema });
```

`src/db/schema/members.schema.ts` (example):
```typescript
import { pgTable, serial, varchar, integer, timestamp } from "drizzle-orm/pg-core";
import { plans } from "./plans.schema";

export const members = pgTable("members", {
  id: serial("id").primaryKey(),
  name: varchar("name", { length: 120 }).notNull(),
  phone: varchar("phone", { length: 30 }).notNull(),
  planId: integer("plan_id").references(() => plans.id),
  status: varchar("status", { length: 20 }).notNull().default("pending"),
  startDate: timestamp("start_date"),
  endDate: timestamp("end_date"),
  registeredAt: timestamp("registered_at").defaultNow().notNull(),
});
```

## 8. Commit convention (senior-dev workflow)

Use **Conventional Commits**, one logical change per commit:

| Prefix | Use for |
|---|---|
| `feat:` | new endpoint, new schema, new capability |
| `fix:` | bug fix |
| `chore:` | tooling, config, deps |
| `refactor:` | code change with no behavior change |
| `test:` | adding/adjusting tests |
| `docs:` | README/guide/comment updates |

Rules:
- One concern per commit (e.g. don't mix "add members schema" with "add payments controller").
- Commit after each working, buildable step — not at the end of a whole resource.
- Push right after each commit (small, frequent pushes, not one big push at the end).
- Use scopes when helpful: `feat(members): add POST /api/members endpoint`.

## 9. Build loop — repeat per resource

For **every** resource (`plans`, `members`, `payments`, `expenses`, `services`, `content`, `notifications`, `reports`), follow this exact loop. Each numbered step below is its own commit:

1. **Plan** — write down the fields, the endpoints (verb + path), and the request/response shape. (No commit needed, just a note/comment — but do this before touching code.)
2. `feat(db): add <resource> table schema` — Drizzle schema file.
3. `feat(<resource>): add zod validators` — `*.validators.ts`.
4. `feat(<resource>): add repository layer` — `*.repository.ts`, pure Drizzle queries.
5. `feat(<resource>): add service layer` — `*.service.ts`, business logic calling the repository.
6. `feat(<resource>): add controller` — `*.controller.ts`, calls the service, shapes responses.
7. `feat(<resource>): add routes + swagger docs` — `*.routes.ts`, mount in `app.ts`.
8. **Test in Swagger UI** — fill the form, hit "Execute", confirm the real DB response. Fix anything found as its own `fix:` commit.
9. `docs: mark <resource> endpoints tested in build guide` (optional, keeps the guide's checklist current).

Only move to the next resource once this one is fully green in Swagger.

### Example: Members — the one with real business logic

`PATCH /api/members/{id}/confirm` activates a pending member and logs a payment in one transaction (member update + `end_date` calculation + payment insert). Because it touches two tables, keep the transaction in the **repository** layer, called from the **service**:

```typescript
// members.repository.ts
export async function confirmMemberTx(id: number, amount: number, methodLabel: string) {
  return db.transaction(async (tx) => {
    const [member] = await tx.select().from(members).where(eq(members.id, id));
    const [plan] = await tx.select().from(plans).where(eq(plans.id, member.planId));

    const endDate = new Date();
    endDate.setDate(endDate.getDate() + plan.durationDays);

    const [updatedMember] = await tx
      .update(members)
      .set({ status: "active", startDate: new Date(), endDate })
      .where(eq(members.id, id))
      .returning();

    const [payment] = await tx
      .insert(payments)
      .values({ memberId: id, amount, methodLabel, date: new Date() })
      .returning();

    return { member: updatedMember, payment };
  });
}
```

Test end-to-end in Swagger: register a pending member (`POST /api/members`) → copy its `id` → call `PATCH /api/members/{id}/confirm` → verify `status: "active"`, a populated `end_date`, and a payment row.

Commit each of the 7 build-loop steps above separately for this endpoint too — don't fold the whole confirm flow into one commit.

## 10. Build order

1. `plans` (needed by members)
2. `members` (incl. confirm endpoint)
3. `payments` (list/read — creation happens via confirm, plus a manual "add payment" endpoint for walk-ins already active)
4. `expenses`
5. `services`
6. `content` (manuals + quotes)
7. `notifications` (schedule CRUD; the actual sending job comes later)
8. `reports` (monthly summary — built last, since it aggregates data from everything above; by now you'll have real test data to verify aggregation against)

## 11. Deploying the backend

Once local testing passes for all endpoints:

1. Push the final state of `main` to `https://github.com/samitad2024/Gympro_backend.git` (already done incrementally if you followed the commit loop).
2. On Render.com: New → Web Service → connect the `Gympro_backend` repo.
3. Build command: `npm install && npm run build`
4. Start command: `npm start`
5. Add environment variables (`DATABASE_URL`, `PORT`) in Render's dashboard.
6. Once deployed, Swagger docs are live at `https://<your-app>.onrender.com/api-docs`.

Update `src/config/swagger.ts` servers array, and commit that change on its own:
```typescript
servers: [
  { url: "http://localhost:4000", description: "Local dev" },
  { url: "https://<your-app>.onrender.com", description: "Production" },
]
```
```bash
git add src/config/swagger.ts
git commit -m "chore: add production server URL to swagger config"
```
